Pipe e FIFO in C 🔄

Comunicazione Inter-Processo (IPC) bidirezionale e unidirezionale.

Pipe e FIFO (Named Pipes)

[Context: Comunicazione Inter-Processo (IPC) bidirezionale o unidirezionale tra processi parent/child o processi scorrelati.]

1. PIPE ANONIME (pipe())

Le pipe anonime vivono solo in memoria e si usano tra processi legati da parentela (es. creati con fork()). Sono unidirezionali.

Creazione e Pattern Base (Fork)

La system call pipe() prende un array di due interi e lo popola con i File Descriptor. L'indice 0 è per la lettura, l'indice 1 è per la scrittura.

int fd[2]; // Array di due interi: fd[0] sarà per LEGGERE, fd[1] per SCRIVERE
// La pipe deve essere creata PRIMA della fork, così il figlio la eredita
if (pipe(fd) == -1) { perror("pipe"); exit(1); }

pid_t pid = fork();

if (pid == 0) { // CHILD (Es. Lettore)
    close(fd[1]); // Chiudo l'estremo di scrittura inutilizzato dal lettore (cruciale!)
    char buf[128];
    // Legge dalla pipe: se è vuota, si blocca finché il padre non scrive o non chiude
    read(fd[0], buf, sizeof(buf));
    close(fd[0]); // Chiude la lettura quando ha finito
    exit(0);
} else {        // PARENT (Es. Scrittore)
    close(fd[0]); // Chiudo l'estremo di lettura inutilizzato dallo scrittore
    char *msg = "Hello";
    // Scrive il messaggio nella pipe. Il +1 serve a includere il terminatore '\0' della stringa
    write(fd[1], msg, strlen(msg) + 1);
    close(fd[1]); // Chiude la scrittura, inviando così un EOF al lettore
    wait(NULL);   // Aspetta che il figlio lettore termini
}

Regola d'oro: Chiudere SEMPRE gli estremi non utilizzati! Se un processo lettore non chiude il suo estremo di scrittura, non riceverà mai l'EOF (End of File) quando lo scrittore finisce.

2. COMPORTAMENTO BLOCCANTE E SEGNALI

Le pipe hanno una capacità limitata (tipicamente 65KB).

Scenario Azione Risultato (Comportamento di Default)
Pipe vuota read() Il processo si blocca in attesa di dati.
Pipe piena write() Il processo si blocca in attesa che il lettore liberi spazio.
Nessun Scrittore (Tutti gli fd[1] chiusi) read() Ritorna 0 (EOF).
Nessun Lettore (Tutti gli fd[0] chiusi) write() Genera il segnale SIGPIPE e termina il processo.

Gestire il SIGPIPE

Se uno scrittore tenta di inviare dati a una pipe in cui tutti i lettori hanno chiuso il canale di lettura, il Kernel invia un segnale SIGPIPE. Di default, questo uccide il processo. Puoi catturarlo con un handler:

#include <signal.h>

// Variabile flag modificabile asincronamente dall'handler (sig_atomic_t garantisce scritture atomiche)
volatile sig_atomic_t sigpipe_received = 0;

// Handler che verrà eseguito automaticamente quando riceviamo il segnale
void handle_sigpipe(int sig) { sigpipe_received = 1; }

// Nel main, prima di eseguire operazioni di scrittura (write) a rischio:
struct sigaction sa = {0}; // Azzera la struttura per evitare memoria sporca
sa.sa_handler = handle_sigpipe; // Associa la nostra funzione
sigaction(SIGPIPE, &sa, NULL); // Sostituisce il comportamento letale di default con il nostro handler

3. PIPE CON NOME / FIFO (mkfifo)

Le FIFO appaiono nel filesystem. Permettono la comunicazione tra processi non legati da parentela. Sono anch'esse unidirezionali.

Creazione e Apertura

Se la FIFO esiste già, mkfifo fallisce con EEXIST, che di solito è un errore ignorabile.

#include <sys/stat.h>
#include <fcntl.h>

// 1. Creazione della FIFO nel FileSystem
// I permessi 0666 permettono lettura e scrittura a tutti (soggetti a umask). 
// EEXIST viene ignorato se un altro processo ha già creato la FIFO
if (mkfifo("/tmp/mia_fifo", 0666) == -1 && errno != EEXIST) {
    perror("mkfifo"); exit(1);
}

// 2. Apertura Bloccante (Client/Processi normali)
// L'apertura in sola lettura (O_RDONLY) si ferma qui (si blocca) finché 
// un altro processo indipendente non aprirà lo stesso file in O_WRONLY
int fd = open("/tmp/mia_fifo", O_RDONLY);

Trucco del Server (O_RDWR)

Spesso i processi Server aprono la FIFO in lettura e scrittura (O_RDWR) anche se intendono solo leggere. Questo trucco impedisce alla open() del server di bloccarsi in attesa del primo client, ed evita che la read() ritorni EOF ogni volta che un singolo client si disconnette.

4. INVIARE STRUTTURE DATI (struct)

Sia le pipe che le FIFO sono flussi di byte. Puoi inviare intere struct C direttamente passandone l'indirizzo e la dimensione.

typedef struct {
    int id;
    char text[128];
} Message;

Message msg_out = {42, "Hello"};
write(fd_write, &msg_out, sizeof(msg_out)); // Scrittura

Message msg_in;
read(fd_read, &msg_in, sizeof(msg_in));     // Lettura

5. TOPOLOGIE MULTIPROCESSO CON PIPE ANONIME

Quando si usano fork() multipli,

APPROFONDIMENTO: Topologie Multiprocesso con Pipe

Quando si generano più processi figli (NCHILDREN), farli comunicare richiede una progettazione attenta dei canali di comunicazione (le pipe) per evitare deadlock (blocchi critici) e assicurare che la lettura restituisca correttamente EOF (End Of File) quando i processi terminano.

1. Topologia ad Anello (Ring Topology)

Nella topologia ad anello, i processi sono disposti in un cerchio logico. Il processo Padre inizia inviando un dato (es. un "token") al Figlio 1. Il Figlio 1 lo elabora e lo passa al Figlio 2, e così via, finché l'ultimo figlio non restituisce il token al Padre.

Regola Matematica: Per N figli (quindi N+1 processi totali, contando il padre), servono esattamente N+1 pipe.

A. Creazione e Struttura
Si crea un array bidimensionale di pipe:

#define NCHILDREN 5
int pipes[NCHILDREN + 1][2];

for (int i = 0; i < NCHILDREN + 1; i++) {
    if (pipe(pipes[i]) == -1) { perror("pipe"); exit(1); }
}

B. Logica di Routing (Chi legge da dove e chi scrive a chi?)
L'indice della pipe definisce il confine tra due processi adiacenti.

  • Il Figlio i (dove i va da 0 a N-1 nel ciclo di fork):
    • Legge dalla pipe che lo collega al predecessore: pipes[i][0]
    • Scrive sulla pipe che lo collega al successore: pipes[i + 1][1]
  • Il Padre:
    • Scrive sulla prima pipe per innescare l'anello: pipes[0][1]
    • Legge dall'ultima pipe per chiudere l'anello: pipes[NCHILDREN][0]

C. Chiusura dei File Descriptor (Cruciale!)
Se non chiudi i file descriptor inutilizzati, i processi si bloccheranno per sempre sulle read(). Ogni processo deve tenere aperti esattamente e solo 2 file descriptor.

Codice del Figlio:

// Chiusura di tutto tranne i 2 estremi necessari al Figlio 'i'
for (int j = 0; j < NCHILDREN + 1; j++) {
    if (j != i)     close(pipes[j][0]); // Chiudo in lettura tutte le pipe tranne la 'i'
    if (j != i + 1) close(pipes[j][1]); // Chiudo in scrittura tutte le pipe tranne la 'i+1'
}

// Lettura, elaborazione e inoltro
int token;
while (read(pipes[i][0], &token, sizeof(token)) > 0) {
    token++;
    write(pipes[i + 1][1], &token, sizeof(token));
}
// Chiusura finale ed exit
close(pipes[i][0]);
close(pipes[i + 1][1]);
_exit(0);

2. Topologia a Stella (Star Topology)

Nella topologia a stella, il processo Padre fa da "Hub" centrale e comunica indipendentemente con ogni singolo figlio. I figli NON comunicano tra di loro. Questo schema è il classico modello Master-Worker.

Regola Matematica: Poiché la pipe anonima è unidirezionale, per avere una comunicazione bidirezionale (Padre <-> Figlio) per N figli, servono 2 * N pipe.

A. Creazione e Struttura
Si usano tipicamente due array separati per chiarezza semantica:

int to_parent[NCHILDREN][2]; // Figlio scrive, Padre legge
int to_child[NCHILDREN][2];  // Padre scrive, Figlio legge

for (int i = 0; i < NCHILDREN; i++) {
    if (pipe(to_parent[i]) == -1 || pipe(to_child[i]) == -1) {
        perror("pipe"); exit(1);
    }
}

B. Logica dei Descrittori nel Figlio
Il figlio di indice i è completamente isolato dagli altri. A lui interessano solo le pipe di indice i.

  • Deve poter scrivere al padre: usa to_parent[i][1]
  • Deve poter leggere dal padre: usa to_child[i][0]

Codice del Figlio:

// 1. Chiusura totale delle pipe degli ALTRI figli
for (int j = 0; j < NCHILDREN; j++) {
    if (j != i) {
        close(to_parent[j][0]); close(to_parent[j][1]);
        close(to_child[j][0]);  close(to_child[j][1]);
    }
}

// 2. Chiusura degli estremi non usati della PROPRIA pipe
close(to_parent[i][0]); // Il figlio non legge dalla pipe diretta al padre
close(to_child[i][1]);  // Il figlio non scrive sulla pipe proveniente dal padre

// 3. Comunicazione
int msg = 42;
write(to_parent[i][1], &msg, sizeof(msg));         // Invia al master
read(to_child[i][0], &msg, sizeof(msg));           // Aspetta risposta

close(to_parent[i][1]);
close(to_child[i][0]);
exit(0);

C. Logica dei Descrittori nel Padre
Il padre (Master) deve mantenere una connessione in lettura e una in scrittura con tutti i figli, ma deve chiudere i propri estremi opposti per non causare deadlock.

Codice del Padre:

// Il padre deve chiudere gli estremi che appartengono ai figli
for (int i = 0; i < NCHILDREN; i++) {
    close(to_parent[i][1]); // Il padre non scrive sulla pipe diretta a se stesso
    close(to_child[i][0]);  // Il padre non legge dalla pipe diretta al figlio
}

// Ora il padre può fare un ciclo per comunicare con tutti
for (int r = 0; r < ROUNDS; r++) {
    for (int i = 0; i < NCHILDREN; i++) {
        int valore;
        // Legge dal figlio i
        read(to_parent[i][0], &valore, sizeof(valore)); 
        
        valore += 1000;
        
        // Risponde al figlio i
        write(to_child[i][1], &valore, sizeof(valore)); 
    }
}

💡 Suggerimento d'Esame: Il Deadlock Silenzioso
Se in una qualsiasi di queste topologie dimentichi un close(pipe[1]) (lato scrittura) nel processo sbagliato, la read() non restituirà mai 0 (EOF) perché il Kernel pensa che ci sia ancora un potenziale scrittore attivo... che in realtà è il processo stesso in ascolto! Quando testi il codice, se i processi rimangono "appesi" indefinitamente alla fine dell'esecuzione, hai dimenticato di chiudere un file descriptor.
