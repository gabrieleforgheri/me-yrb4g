Processi (fork), wait e Memoria

[Context: Creazione di processi, gerarchie, duplicazione della memoria (Copy-On-Write) e condivisione dei file descriptor in ambiente Unix/Linux.]

1. CREAZIONE DI PROCESSI (fork)

La system call fork() crea un nuovo processo duplicando il processo chiamante.

  • Valori di ritorno della fork():
    • < 0: Errore (la fork è fallita).
    • == 0: Siamo nel Processo Figlio (Child).
    • > 0: Siamo nel Processo Padre (Parent). Il valore restituito è il PID (Process ID) del figlio appena creato.
  • Funzioni utili per gli ID:
    • getpid(): Restituisce il PID del processo corrente.
    • getppid(): Restituisce il PID del processo padre (Parent PID).
// Creazione del nuovo processo: da questo punto il codice si biforca
pid_t pid = fork();

if (pid < 0) {
    // La fork ha fallito (es. troppi processi nel sistema)
    perror("fork failed");
    exit(1);
} else if (pid == 0) {
    // CODICE DEL FIGLIO: la fork() restituisce 0 al processo appena clonato
    printf("Sono il figlio, il mio PID è %d\n", getpid());
    exit(0); // Il figlio termina qui
} else {
    // CODICE DEL PADRE: la fork() restituisce il PID del nuovo figlio generato
    printf("Sono il padre, il PID di mio figlio è %d\n", pid);
}

Fork Multiple

Se esegui chiamate sequenziali a fork() senza ramificazioni if, il numero di processi cresce esponenzialmente (2n). Ad esempio, 3 fork() consecutive creano in totale 8 processi (1 padre originale + 7 figli/nipoti).

2. SINCRONIZZAZIONE PADRE-FIGLIO (wait)

Un processo padre usa wait o waitpid per sospendere la propria esecuzione finché un processo figlio non termina.

  • wait(NULL): Aspetta la terminazione di un qualsiasi processo figlio.
  • waitpid(pid, &status, options): Aspetta la terminazione di uno specifico figlio indicato dal pid.

Lettura dello Status

Quando il figlio termina tramite exit(valore), il padre può leggere quel valore decodificando la variabile status con delle macro specifiche:

int status; // Variabile usata per memorizzare l'esito della terminazione
// Il padre si blocca e attende la fine dello specifico figlio
waitpid(pid_figlio, &status, 0);

// Macro per verificare se il figlio ha usato exit() o return normalmente
if (WIFEXITED(status)) {
    // Estrae il codice di uscita effettivo (es. lo 0 di exit(0))
    int valore_ritorno = WEXITSTATUS(status);
    printf("Il figlio ha restituito: %d\n", valore_ritorno);
} else {
    // Entra qui se il figlio è stato ucciso da un segnale (es. SIGKILL, segfault)
}

3. MEMORIA E VARIABILI: COPY-ON-WRITE

Dopo una fork(), il figlio ottiene una copia esatta della memoria del padre.

  • Variabili Locali e Globali: Le modifiche fatte dal figlio alle variabili (es. x += 5) NON si riflettono sul padre, e viceversa.
  • Memoria Heap (malloc): Anche l'allocazione dinamica viene duplicata. Il figlio vedrà la variabile puntata allo stesso indirizzo virtuale del padre.
  • Copy-On-Write (Ottimizzazione del Kernel): Subito dopo la fork(), padre e figlio condividono la stessa pagina di memoria fisica (PFN - Page Frame Number). La vera duplicazione in RAM avviene solo quando uno dei due processi tenta di scrivere/modificare quel dato.

Regola d'oro: Se allochi con malloc prima della fork, sia il padre che il figlio dovranno chiamare free() sui rispettivi puntatori per non avere memory leak.

4. CONDIVISIONE DEI FILE DESCRIPTOR

Se il padre apre un file (usando open) prima di chiamare la fork(), i descrittori vengono condivisi tra padre e figlio. Questo ha effetti critici:

  • Scritture Miste: Se padre e figlio scrivono contemporaneamente nello stesso descrittore condiviso, i caratteri si mescoleranno in modo imprevedibile nel file (interleaving).
  • Offset Condiviso (Cruciale): Padre e figlio condividono il "cursore" (offset) all'interno del file. Se il File Descriptor viene clonato e il Figlio 1 legge 1 byte dal file, il cursore avanza. Quando il Figlio 2 andrà a leggere dal file, leggerà il byte successivo, non il primo.
// Apre un file prima della fork: il File Descriptor verrà condiviso!
int fd = open("data.txt", O_RDWR); 
write(fd, "ABCDEF", 6); // Scrive 6 byte, l'offset ora è a 6
lseek(fd, 0, SEEK_SET); // Riporta l'offset (cursore) all'inizio (0)

pid_t c1 = fork();
if (c1 == 0) {
    char ch;
    // Figlio 1 legge 1 byte ('A'). L'offset condiviso avanza a 1 per TUTTI
    read(fd, &ch, 1); 
    exit(0);
}

wait(NULL); // Il padre aspetta che il Figlio 1 termini

pid_t c2 = fork();
if (c2 == 0) {
    char ch;
    // Figlio 2 legge 1 byte ('B'). L'offset non parte da 0, era stato spostato da Figlio 1!
    read(fd, &ch, 1); 
    exit(0);
}

APPROFONDIMENTO: La Famiglia exec e il Pattern Fork-Exec

La system call fork() crea un clone esatto del processo padre. Ma se vogliamo eseguire un programma completamente diverso (es. lanciare ls o Firefox), dobbiamo usare la famiglia di funzioni exec.

La funzione exec sostituisce completamente l'immagine in memoria del processo chiamante con un nuovo programma.

  • Non crea un nuovo processo: Il PID rimane identico!
  • Non ritorna mai: Se exec ha successo, il codice successivo non verrà MAI eseguito, perché il programma originale è stato spazzato via dalla RAM. Ritorna -1 solo se fallisce (es. comando non trovato).

La Nomenclatura (I Suffissi di exec)

Esistono diverse varianti di exec, distinguibili dalle lettere finali:

  • l (List): Gli argomenti vengono passati come una lista separata da virgole. L'ultimo argomento deve sempre essere NULL.
  • v (Vector): Gli argomenti vengono passati come un array di stringhe (char *argv[]).
  • p (Path): Usa la variabile d'ambiente $PATH del sistema operativo per cercare l'eseguibile (non serve scrivere /bin/ls, basta ls).
  • e (Environment): Permette di passare un array customizzato di variabili d'ambiente.

Esempi Pratici (Pattern Fork-Exec)

Normalmente si fa una fork() e, solo nel figlio, si chiama la exec:

1. Uso di execlp (Lista + Path):

pid_t pid = fork();
if (pid == 0) {
    // Arg 0 è per convenzione il nome del programma stesso
    execlp("ls", "ls", "-l", "-a", "/tmp", NULL); 
    perror("exec fallita"); // Questa riga viene eseguita SOLO se execlp fallisce!
    exit(1);
}
wait(NULL); // Il padre aspetta che 'ls' finisca

2. Uso di execv (Vettore, Path Assoluto):

pid_t pid = fork();
if (pid == 0) {
    char *args[] = {"/bin/cat", "miofile.txt", NULL}; // Array terminato da NULL
    execv("/bin/cat", args);
    perror("exec fallita");
    exit(1);
}

I "WEIRD BEHAVIOURS": Corner Cases di fork ed I/O

I file di test che hai fornito nascondono le "trappole" più comuni che capitano nei laboratori e agli esami di Sistemi Operativi.

1. La Trappola dei Buffer I/O non svuotati (Il caso printf)

Guarda questo codice:

printf("Hello"); // SENZA \n
fork();

Cosa succede? La funzione printf della libreria standard C (User Space) usa dei buffer. Se non metti il carattere di a capo \n (o non chiami fflush(stdout)), la stringa "Hello" rimane bloccata nel buffer in RAM, non viene ancora scritta sul terminale.

Quando chiami fork(), il Kernel clona interamente la RAM, copiando anche il buffer pieno di "Hello" nel processo figlio! Alla fine del programma, entrambi i processi (padre e figlio) svuoteranno i loro rispettivi buffer, stampando "HelloHello".

2. exit(0) vs _exit(0)

Per risolvere il problema dei buffer nei figli, bisogna conoscere la differenza tra le funzioni di uscita:

  • exit(0) (Libreria C): Esegue le procedure di pulizia, svuota tutti i buffer I/O pendenti (stampando i dati rimasti appesi) e poi termina.
  • _exit(0) (System Call diretta): Chiede al Kernel di abbattere immediatamente il processo. I buffer in User Space NON vengono svuotati e i dati pendenti vanno persi.

Regola d'oro: Quando un processo figlio termina dopo aver fatto il suo lavoro interno o se fallisce una exec, è spesso consigliato usare _exit() per evitare di flussare accidentalmente due volte i buffer aperti dal padre.

3. File Descriptors: Aperti PRIMA o DOPO la Fork?

Questa è una domanda da 30 e lode.

  • Apertura PRIMA della fork: Se il padre fa open() e poi fork(), padre e figlio condividono la stessa voce nella tabella dei file aperti del Kernel.
    L'offset (cursore) è condiviso. Se il figlio legge 5 byte o fa una lseek spostando il cursore a +5, quando il padre andrà a leggere, partirà dal sesto byte! Questo può causare letture mischiate (interleaving).
  • Apertura DOPO la fork: Se il padre fa fork() e poi sia il padre che il figlio fanno open("file.txt"), avranno file descriptor indipendenti che puntano a strutture separate nel Kernel. Ognuno avrà il suo cursore (offset) a 0. Se leggono, leggeranno entrambi dall'inizio del file senza disturbarsi.

4. L'Illusione della Memoria e il Copy-On-Write (CoW)

Guarda questo caso con la memoria Heap:

int *p = malloc(sizeof(int));
*p = 100;
if (fork() == 0) {
    *p = 111;
    printf("Figlio: *p = %d, address = %p\n", *p, p);
} else {
    printf("Padre: *p = %d, address = %p\n", *p, p);
}

Il risultato strano: Entrambi i processi stamperanno lo stesso identico indirizzo di memoria (es. 0x55d78), ma il valore letto sarà diverso (111 nel figlio, 100 nel padre).

Perché succede? Linux gestisce la memoria tramite la Virtual Memory. malloc restituisce un Indirizzo Virtuale, che è stato clonato dalla fork.
Tuttavia, quando il figlio tenta di modificare la variabile (*p = 111), la CPU genera una trap. Il Kernel interviene col meccanismo di Copy-On-Write (CoW): copia silenziosamente la pagina fisica della RAM in un'altra zona e reindirizza il puntatore virtuale del figlio verso la nuova pagina.

Padre e figlio vedono lo stesso indirizzo virtuale, ma fisicamente (il Page Frame Number - PFN) stanno puntando a due chip di RAM diversi!
