1. Il Flusso di una Socket Stream (Modello Server-Client)

Per capire come i tuoi due programmi si connettono, immagina una telefonata: il server installa il telefono e aspetta la chiamata; il client compone il numero.

Lato Server (Il Ricevente)

Il server deve preparare l'endpoint e mettersi in ascolto. I passaggi chiave nel codice sono:

socket(): Crea l'endpoint di comunicazione.

srv_fd = socket(AF_UNIX, SOCK_STREAM, 0);

bind(): Associa la socket a un indirizzo specifico (in questo caso, il percorso sul file system). Prima di fare il bind, si usa unlink() per rimuovere eventuali file socket residui da esecuzioni precedenti.

bind(srv_fd, (struct sockaddr *)&srv_addr, sizeof(srv_addr));

listen(): Trasforma la socket in una socket passiva, pronta ad accettare le connessioni, definendo una coda massima di client in attesa.

listen(srv_fd, 2);

accept(): Questa chiamata è bloccante. Il server si ferma finché un client non avvia una connessione. Quando il client si connette, accept() estrae la connessione dalla coda e restituisce un nuovo file descriptor (es. ci1.fd). Questo nuovo FD verrà usato per comunicare esclusivamente con quel determinato client, lasciando srv_fd libero di ascoltare altre chiamate.

ci1.fd = accept(srv_fd, NULL, NULL);

Lato Client (Il Chiamante)

Il client ha un flusso molto più snello:

socket(): Crea la sua socket.

connect(): Avvia la richiesta di connessione verso l'indirizzo (il file del socket) su cui il server è in ascolto. Anche questa operazione è bloccante finché il server non esegue la accept().

connect(sock_fd, (struct sockaddr *)&srv_addr, sizeof(srv_addr));

2. Lettura e Scrittura: Gestire il Flusso di Byte

Nelle socket di tipo SOCK_STREAM, i dati sono inviati come un flusso continuo di byte. Questo significa che se invii una struttura di 4 byte, la read() potrebbe riceverne 2 adesso e 2 al prossimo ciclo.

Per risolvere questo problema, nel tuo codice sono state implementate le funzioni helper write_full e read_full. Sono utilissime per l'esame perché gestiscono due problemi sistemistici reali:

  • Letture/Scritture parziali: Il ciclo while continua finché non sono stati trasferiti esattamente tutti i byte richiesti (count).
  • Interruzioni da segnali (EINTR): Se una chiamata di sistema viene interrotta da un segnale (come il SIGUSR1 usato nel gioco), errno viene impostato a EINTR. Il codice usa continue per riprendere l'operazione interrotta senza far fallire il programma.

CHEATSHEET: I/O e File Control in C

[Context: Gestione FIFO, pipe, read/write, poll() e I/O non bloccante in ambiente Unix/Linux]

1. APERTURA FILE (open)

Restituisce un File Descriptor (FD) intero. Se fallisce, ritorna -1.

  • Bloccante (Default):
    int fd = open("/tmp/fifo", O_RDONLY);
  • Non Bloccante:
    int fd = open("/tmp/fifo", O_RDONLY | O_NONBLOCK);

2. READ & WRITE SU FIFO

Sintassi base: ssize_t n = read(fd, &buf, sizeof(buf));

Scenario Bloccante Non Bloccante
Read: Dati presenti Ritorna byte letti Ritorna byte letti
Read: Niente dati Si blocca in attesa Ritorna -1, errno=EAGAIN
Read: Nessun writer Ritorna 0 (EOF) Ritorna -1, poll genera POLLHUP
Write: Buffer pieno Si blocca in attesa Ritorna -1, errno=EAGAIN

3. MULTIPLEXING CON POLL()

Evita il busy waiting su file non bloccanti.

Struttura e Inizializzazione:

#include <poll.h>
struct pollfd poll_set[NUM_FILE];

for (int i = 0; i < NUM_FILE; i++) {
    poll_set[i].fd = fds[i];     // FD da monitorare
    poll_set[i].events = POLLIN; // Eventi da ascoltare
    poll_set[i].revents = 0;     // Output del kernel
}

B. Ciclo Principale e Gestione Eventi

La system call poll si blocca finché non accade qualcosa. Controllare gli eventi richiede l'uso dell'operatore Bitwise AND (&), mai dell'uguaglianza (==).

// timeout -1 significa attesa infinita
int pronti = poll(poll_set, NUM_FILE, -1); 
if (pronti < 0) {
    perror("Errore poll");
    exit(1);
}

for (int i = 0; i < NUM_FILE; i++) {
    if (poll_set[i].fd == -1) continue; // Salta FD chiusi

    // 1. Controllo Disconnessione (Hang Up)
    // Usa & per verificare se il bit POLLHUP è acceso tra i vari eventi
    if (poll_set[i].revents & POLLHUP) {
        close(poll_set[i].fd);
        poll_set[i].fd = -1;
        // Aggiorna contatore chiusure, es: nclosed++;
        continue;
    }

    // 2. Controllo Dati Pronti
    if (poll_set[i].revents & POLLIN) {
        ssize_t nread;
        int value;
        
        // Loop per svuotare completamente il buffer non bloccante
        while ((nread = read(poll_set[i].fd, &value, sizeof(value))) > 0) {
            // Elabora 'value' (es. accumulator += value;)
        }
        
        // Verifica se l'uscita dal while è un VERO errore o solo fine dati
        if (nread == -1 && errno != EAGAIN) {
            perror("Errore in read");
            exit(1);
        }
    }
}
