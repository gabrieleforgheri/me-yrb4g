CHEATSHEET SISTEMI OPERATIVI: Paradigmi di I/O, Asincronia e Descrittori

Un'analisi architettonica profonda su come il Kernel Linux gestisce l'input/output e i descrittori di file. Da un approccio sequenziale bloccante fino alle architetture asincrone event-driven.

1. IL COLLO DI BOTTIGLIA: I/O Bloccante (Synchronous Blocking)

Nei classici server e client basati su pipe o FIFO (es. blocking-server.c e 02a-blocking-read.c), vediamo il comportamento di default di Unix.

La logica: Quando il server chiama read(fds[i], ...) su una FIFO vuota, il Kernel "addormenta" il processo. Lo toglie dalla coda di esecuzione della CPU e lo mette in attesa finché non arrivano dati.

  • Vantaggio: Il codice è banale da scrivere. Il processo consuma 0% CPU mentre aspetta.
  • Il grave difetto architetturale: Se il Server deve gestire 1000 client e si blocca in attesa del Client 0, i Client dall'1 al 999 rimarranno bloccati a loro volta, anche se hanno già inviato i loro dati. È un design strettamente sequenziale che distrugge la concorrenza.

Apertura e Comportamento Default

L'apertura standard restituisce un File Descriptor (FD) intero (o -1 in caso di errore):

int fd = open("/tmp/fifo", O_RDONLY);

In questo scenario, una read() su un canale vuoto o una write() su un buffer pieno bloccano irreversibilmente il processo fino allo sblocco della risorsa.

2. LA SOLUZIONE ATTIVA: I/O Non Bloccante (Synchronous Non-Blocking)

L'ingegneria del software qui cambia: imponiamo al Kernel di non addormentare mai il processo. Se non ci sono dati, la system call deve fallire "gentilmente" e restituire subito il controllo al programma.

Come si trasforma un file in Non-Bloccante: fcntl

Se apriamo un file da zero possiamo usare O_NONBLOCK direttamente nella open. Se il descrittore è già stato aperto o ereditato (es. Standard Input), alteriamo le proprietà a runtime usando fcntl (File Control). L'operazione bit a bit | (OR) è fondamentale per aggiungere un flag senza sovrascrivere quelli già esistenti.

#include <fcntl.h>
#include <errno.h>

// 1. Legge la maschera dei flag correnti dal Kernel
int flags = fcntl(fd, F_GETFL); 

// 2. Riscrive la maschera aggiungendo O_NONBLOCK
fcntl(fd, F_SETFL, flags | O_NONBLOCK);

Il significato di EAGAIN

In modalità non bloccante, una read() vuota o una write() su una pipe piena restituiscono -1. Per distinguere un vero errore hardware da un semplice "riprova più tardi", l'ingegnere deve controllare la variabile globale errno:

ssize_t n = read(fd, buf, sizeof(buf));
if (n < 0 && (errno == EAGAIN || errno == EWOULDBLOCK)) {
    // Non è un errore critico. Il buffer è solo vuoto/pieno in questo microsecondo.
    // Il programma può passare a interrogare il client successivo.
}

Multiplexing con poll() per evitare il busy-waiting

Anche se il file è non-bloccante, interrogare continuamente in loop migliaia di descrittori (polling attivo) spreca cicli CPU. La soluzione è usare poll() per delegare al kernel l'ascolto passivo su array di descrittori.

#include <poll.h>
struct pollfd poll_set[NUM_FILE];

for (int i = 0; i < NUM_FILE; i++) {
    poll_set[i].fd = fds[i];     // FD da monitorare
    poll_set[i].events = POLLIN; // Eventi da ascoltare
    poll_set[i].revents = 0;     // Output del kernel
}

// -1 = si addormenta finché almeno UNO non ha eventi
int ready = poll(poll_set, NUM_FILE, -1); 

// Loop per trovare chi ha attivato la poll
for (int i = 0; i < NUM_FILE; i++) {
    if (poll_set[i].revents & POLLIN) {
        // Dati pronti: leggiamo finché il buffer non bloccante non si svuota
        while ((nread = read(poll_set[i].fd, &value, sizeof(value))) > 0) { /* usa i dati */ }
    }
}

3. L'INVERSIONE DEL CONTROLLO: I/O Asincrono guidato dai Segnali

Fare polling con poll() risolve molti problemi, ma vincola ancora il processo a dover esplicitamente verificare l'array. L'approccio Event-Driven realizza l'inversione del controllo: non sei tu a chiedere al Kernel se ci sono dati, è il Kernel che interrompe il tuo programma in background inviandoti un segnale SIGIO quando i dati sono pronti.

Il Setup Architetturale per l'Asincronia:

Affinché questo meccanismo funzioni, bisogna "istruire" il Kernel con 3 passaggi precisi:

  1. Gestione del Segnale: Dire al processo cosa fare quando riceve l'interruzione. 
    struct sigaction sa = {0};
sa.sa_handler = sigio_handler; // Funzione che imposta un flag (es. got_sigio = 1)
sigaction(SIGIO, &sa, NULL);
  2. Proprietà del Descrittore (F_SETOWN): Un file potrebbe essere condiviso. Dobbiamo dire al Kernel esattamente a quale Process ID (PID) inviare il segnale SIGIO quando questo specifico FD riceve dati. 
    fcntl(fd, F_SETOWN, getpid());
  3. Attivazione Flag (O_ASYNC): Abilitare fisicamente la generazione asincrona dei segnali per quel file. 
    int flags = fcntl(fd, F_GETFL);
fcntl(fd, F_SETFL, flags | O_ASYNC | O_NONBLOCK);

Nota didattica: Si abbina spesso O_ASYNC a O_NONBLOCK. Quando arriva il SIGIO, il programma salta nell'handler, alza la bandierina e poi avvia un ciclo while(read(...) > 0) per svuotare l'intero buffer senza rischiare di bloccarsi se legge un byte di troppo.

4. MANIPOLAZIONE AVANZATA A LIVELLO DI KERNEL

Redirezione dei Canali: dup2

La system call dup2(oldfd, newfd) è il cuore di come la Shell gestisce operatori come >. Clona un descrittore in una posizione specifica della tabella dei file del processo.

int logfd = open("output.log", O_WRONLY | O_CREAT | O_TRUNC, 0644);

// Chiude automaticamente lo standard output (1) e lo ricollega a logfd
dup2(logfd, 1); 
close(logfd); // Pulizia: il vecchio fd non serve più, ora logfd e 1 sono la stessa cosa

printf("Questo va nel file, non a schermo!\n");

Ottimizzazione Spaziale: File Sparsi (Sparse Files)

Un "Sparse File" sfrutta un'illusione del file system. Usa lseek() per muovere il cursore logico di scrittura molto in avanti rispetto ai dati fisicamente scritti, lasciando un "buco".

// Salta in avanti di 1 GB senza scrivere nulla
off_t offset = lseek(fd, 1024LL * 1024 * 1024, SEEK_SET);

// Scrivendo un solo byte alla fine, il sistema "crede" che il file sia da 1GB.
write(fd, "A", 1);

Spiegazione Logica: Il Kernel Linux è intelligente. Per i blocchi di zeri logici saltati con lseek, non alloca veri blocchi sui settori del disco fisso (risparmiando usura e spazio). Se esegui ls -l vedrai 1GB (dimensione logica), ma se usi du -h (Disk Usage) vedrai che occupa solo 4 Kilobyte reali sul disco fisico!

Advisory File Locking (struct flock)

Il locking tramite fcntl() è di tipo advisory (cooperativo). Significa che il Kernel non impedisce fisicamente a un processo malintenzionato di scrivere sul file; spetta ai programmi concorrenti cooperare richiedendo esplicitamente il lock.

struct flock lock = {0};
lock.l_type   = F_WRLCK;   // F_RDLCK (Condiviso), F_WRLCK (Esclusivo), F_UNLCK (Sblocca)
lock.l_whence = SEEK_SET;  // Partiamo dall'inizio del file

// F_SETLK tenta in modo NON bloccante. Se occupato, ritorna -1 e errno=EAGAIN
if (fcntl(fd, F_SETLK, &lock) == -1) {
    perror("Impossibile bloccare, file già occupato");
} else {
    // Sezione critica
    lock.l_type = F_UNLCK;
    fcntl(fd, F_SETLK, &lock);
}
